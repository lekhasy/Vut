# Decision Log — Velucid PRD (2026-06-07 redesign)

Captures every significant decision, change, and override made during this PRD session. Decisions land here as they're made, not at finalize.

## Conventions

- Date in ISO 8601.
- Owner is the person who made the call.
- Status: `proposed` → `accepted` → `superseded` / `reverted`.

## Decisions

### 2026-06-07 — Intent: Create fresh PRD, replacing the obsolete one

- **Context.** Old `PRD.md` (repo root) assumed tag-based organization, Auth0, kanban, and a forecast-as-hero product. The tag model was rejected: it loses the *meaning* of actions in the event log (a `TagAdded` event is just a tag flip, not a structurally legible "this item was assigned to team X"). Forecast is being re-evaluated — see Open Questions.
- **Decision.** Throw out old PRD. Build a fresh one. New model: explicit aggregates (Workspace, User, Account, Initiative, Team, Member, WorkItem, Label), priority-list UX (no kanban), email-OTP auth (no Auth0).
- **Status.** accepted
- **Owner.** Syle
- **Wipe scope.** Stories, epics, and prior change proposals removed; `architecture.md` kept (to be edited after PRD); `sprint-status.yaml` reset.

### 2026-06-07 — Auth: drop Auth0 entirely; own email-OTP login

- **Context.** Old PRD had Auth0 + GitHub SSO + internal email verification. The new model wants zero third-party identity provider: user enters email, gets a one-time code by email, enters the code, session created.
- **Decision.** Remove Auth0 dependency from the stack. Astro web app (apps/web) remains the sole auth entry point. Need to add: transactional email provider (TBD), short-lived code store (TTL'd), session management.
- **Status.** accepted
- **Owner.** Syle
- **Open sub-decisions.** Email provider choice (Resend / SES / Postmark); session model (server-side session vs signed cookie); code TTL; rate limiting strategy. Surface in Open Questions and resolve during Drafting.

### 2026-06-07 — UX: priority list with drag-to-reorder, no kanban

- **Context.** Old PRD had kanban board with columns and drag-between-columns. The new model wants no kanban.
- **Decision.** No kanban boards, no columns, no drag-between-columns. Each Team has one WorkItem list. The list is **drag-to-reorder**; the highest-priority item sits at the top. Drag is therefore in — just on a list, not on cards across columns.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** "Drag-and-drop" in the UX guidance is no longer a kanban affordance; it's a list-reorder affordance. Update §4.7 / §7.3 of the old PRD reference (and any UX doc) accordingly.

### 2026-06-07 — Projector: keep C# projector; defer TS port

- **Context.** Workspace was mid-migration from .NET projector to TS/Bun projector (Epic 4 of prior sprint). TS port is not finished and the user no longer wants to invest there for now.
- **Decision.** C# (.NET) projector is the projector for this PRD. TS/Bun port is **deferred indefinitely** — not in scope for current planning.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** `architecture.md` needs a major edit: the "porting from .NET to TS" framing is wrong, the TS projector sub-project goes on the shelf, and the read-model/event-store topology should be redrawn around the .NET projector. Flag for the architecture edit pass after the PRD.

### 2026-06-07 — Surface split: Astro (BFF + landing) vs React SPA (local-first app)

- **Context.** Astro is currently both the BFF and the SSR shell for the whole product. The new model wants local-first for the app, which is incompatible with Astro's per-request SSR.
- **Decision.**
  - **Astro 5** = sole auth entry point + landing page only. Still the BFF, but serves no app content.
  - **React SPA** = the product surface. Local-first. Lives in a separate Nx app (e.g., `apps/app` or `apps/local` — name TBD in architecture).
- **Status.** accepted
- **Owner.** Syle
- **Implication.** The current `apps/web` Astro app gets split. The React 19 SPA (already partly built) becomes the product surface and gains a local store (IndexedDB or equivalent — architecture decision). Architecture doc needs to redraw the request flow: React SPA ↔ Astro BFF ↔ silo, with the SPA also holding a local projection.
- **PRD scope note.** PRD describes the *behavior* (instant reads, optimistic writes, sync in the background) without committing to a specific local store or sync protocol — those are architecture calls.

### 2026-06-07 — PRD scope: product behavior, not tech stack

- **Context.** Old PRD mixed product FRs with stack details (Auth0, Bun, K3s, etc.).
- **Decision.** This PRD describes *what* the product does and *what behavior* the user sees. Stack, transport, and infra choices live in `architecture.md`. Where product-visible behavior depends on a tech choice (e.g., "reads feel local" implies a local store), the PRD names the behavior and references the architecture for the mechanism.
- **Status.** accepted
- **Owner.** Syle

### 2026-06-07 — Local-first behavior: snappy, not offline; real-time teammates (transport TBD)

- **Context.** Two readings of "local first": (a) snappy on a good network via a local store, or (b) full offline-capable with conflict-merge later. Need to pick one for MVP.
- **Decision.** **Option (a)**. Local store for instant reads and optimistic writes. **No offline mode** — we don't want users to make changes offline and merge later. Teammate changes are visible **in real time** (transport — WebSocket or equivalent — is TBD; the real-time client lib is still in development, so the PRD will not commit to a transport). The PRD describes the *behavior*; architecture picks the mechanism.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** Local store (architecture) caches the read model. Writes go through the BFF to the silo. Teammate updates push into the local store as they happen, so the UI re-renders live. PRD uses "real-time" without naming WebSocket/transport.

### 2026-06-07 — Notifications: email-only in MVP

- **Context.** Need to decide notification channel for assignment, mention, invitation, etc.
- **Decision.** **Email only in MVP.** No in-app notification center, no notification bell, no unread counts. The transactional email provider chosen for OTP codes is reused for notification emails.
- **Status.** accepted
- **Owner.** Syle

### 2026-06-07 — New aggregate: WorkspaceInvitation (one stream per invitation, not per workspace)

- **Context.** Need a way to add new Accounts to a Workspace. Several shapes possible; user picked the "invite by email with link/code" path.
- **Decision.** New aggregate: **WorkspaceInvitation**. It has its own lifecycle (Pending → Accepted / Revoked / Expired) and its **own event stream — one stream per invitation**, named `workspace-invitation-{uniqueInvitationId}`. Invitations do *not* show up in the Workspace's event stream, so the workspace history stays focused on the workspace's actual evolution. Owner flow: "Invite people" CTA → enter email → choose one or more Teams (team assignment is optional) → click Invite → invitation email goes out with a one-time accept link/code → invitee signs in (or signs up via email-OTP) → on accept, the Account is created and (optionally) the TeamMember record(s) are created for the chosen Teams.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** This is the 9th aggregate in the data model. PRD §3 Glossary must list and define it. PRD §4 must cover the invite/accept lifecycle end-to-end. Architecture: each invitation grain/stream is independent — projector can ignore the whole invitation stream family for workspace read models and surface it only in the invitations view. Accept binds to User + Account + TeamMember creation.

### 2026-06-07 — Forecast is in MVP and is initiative-based (not tag-based), with two independent flavors

- **Context.** Old PRD had forecasting as the hero, parameterized by tags. The new model has no tags as filters — Initiatives are the unit of strategic work, so they become the forecast dimension. Without forecasting, there's no reason for anyone to pick Velucid over Linear.
- **Decision.** **Forecast is in MVP.** Two independent forecasts, neither relies on the other:
  1. **Per-Initiative forecast** — overall completion curve for the Initiative's WorkItems.
  2. **Per-Initiative × Per-Team forecast** — answers "which team is the bottleneck / who came last" by forecasting each Team's contribution to the Initiative.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** This is the strategic differentiator and must be present in MVP. `docs/velucid_forecasting_spec.md` is **not updated in this PRD pass** — it stays as-is, and the forecast work is sequenced **last in the build order** (after the rest of MVP is in). PRD describes the two forecast flavors and their independence at a product level; the math/algorithm is in the forecasting spec, owned by architecture.
- **Note for the spec.** When the spec is eventually re-scoped, the old tag-based filter dimension is gone; the new dimension is Initiative (and Team as the cross-cut). The 7-day minimum history, Monte Carlo, S-curve, and stat-card UX all carry over.

### 2026-06-07 — WorkItem status: per-Team, six categories, defaults only in MVP

- **Context.** Status definitions need a home. Per-product was the old model; the new aggregates are Team, Initiative, Workspace.
- **Decision.** **Statuses are defined per Team.** Six **Status Categories** exist; each has one default Status:
  - **Backlog** — default status: `Backlog` (work not ready to start)
  - **Unstarted** — default status: `Todo` (ready to start)
  - **Started** — default status: `InProgress`
  - **Completed** — default status: `Done`
  - **Canceled** — default status: `Canceled`
  - **Duplicate** — default status: `Duplicate`
- Each WorkItem's Status belongs to exactly one Category. In MVP, **only the default six statuses exist**. The owner-of-the-Team-can-add-new-statuses-to-a-category capability is **deferred** (e.g., a future "Blocked" status under the Backlog category) — noted here, not built now.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** PRD §3 Glossary needs `Status`, `Status Category`, and the six default statuses. §4 covers status transitions within and between categories (the "drag a card from Backlog to Todo to InProgress" mental model). The deferred extension point must be called out so architecture doesn't paint itself into a corner.

### 2026-06-07 — Roles: flat — Owner and Member only, with explicit Owner-only actions

- **Context.** Need a workspace role model.
- **Decision.** **Two roles only: `Owner` and `Member`.** Flat hierarchy. Members can see basically everything in the workspace (the read model is shared). **Owner-only actions** (in MVP):
  - Workspace configuration (name, etc.)
  - Team configuration
  - Initiative creation
- Everything else (creating work items, assigning, moving, commenting) is available to any Member.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** PRD §3 Glossary needs `Owner` and `Member` (role). §4's FRs call out Owner-only capabilities explicitly. Authorization model (OpenFGA tuples) is in architecture — the PRD just names the role and the actions gated by it. **Terminology clash to resolve in §3 Glossary:** the *join* of Account + Team is currently called "Member" in the aggregate list; the *workspace role* is also "Member". I propose renaming the join to `TeamMember` to keep the role clean. Flag for user confirmation in the Glossary draft.

### 2026-06-07 — Onboarding: first workspace OR enter invite code

- **Context.** New-user empty state needed for §6 User Journeys and §4 FRs.
- **Decision.** Two distinct onboarding paths, picked at first sign-in:
  1. **No pending invitation:** the user is prompted to **create their first Workspace** by giving it a **globally unique name**. (Globally unique = no other Workspace in Velucid has that name — across all tenants. This protects vanity / brand names.)
  2. **Pending invitation present:** the user is prompted to **enter the one-time invite code** sent to their email. Entering the correct code **jumps them directly into the Workspace** they were invited to. (The Account is created in that Workspace on accept; no "create your first workspace" prompt is shown in this case.)
- **Status.** accepted
- **Owner.** Syle
- **Implication.** Onboarding UX has two mutually-exclusive paths. PRD §4 needs both flows as User Journeys and FRs. The globally-unique-workspace-name rule is an FR — needs server-side uniqueness check + a clean error UX.

### 2026-06-07 — No WorkItem "archive" — `Canceled` is a status, not a separate action

- **Context.** Asked rhetorically whether to include a "Member archives a WorkItem" UJ. The user confirmed: no archive concept. WorkItem "removal" from active work is just a status transition to `Canceled` (one of the six default Status Categories).
- **Decision.** **No archive feature.** WorkItems have no special archive action in MVP. To stop work, change the WorkItem's status to `Canceled`. The six default Status Categories (incl. `Canceled`) cover it.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** §4.8 WorkItems does not include any archive / soft-delete capability. §3 Glossary needs no changes. (WorkItem *deletion* as a hard remove is a separate question — also out of scope for MVP unless called out later.)

### 2026-06-07 — Build order within MVP: auth → realtime + local-first → structure → WorkItems → Labels → Forecast

- **Context.** PRD says *what* the product does, not *when* it's built. But the user wants explicit sequencing to drive sprint planning and avoid building things on a fragile foundation.
- **Decision.** Build order within MVP:
  1. **Authentication & session** (§4.1) — email-OTP, session, the BFF owns login.
  2. **Realtime sync + local-first** (§4.10) — **immediately after auth, on the client side.** No structural features ship until the React SPA can read from a local store and propagate changes (in both directions) in real time. The user wants this foundation in place before any structural feature is built on top of it.
  3. **Workspace, Account, Team, TeamMember, Initiative, WorkspaceInvitation** (§4.2–§4.6, §4.3) — the structural model.
  4. **WorkItems** (§4.8) — with statuses, priority, team assignment. Built before Labels per explicit user call.
  5. **Labels** (§4.7) — workspace-scoped tags. **Deliberately deprioritized** — do not build before WorkItems.
  6. **Forecast** (§4.9) — **last in MVP**, per the earlier forecasting decision. The math is in `docs/velucid_forecasting_spec.md` and is unchanged in this PRD pass.
  7. **Email notifications** (§4.11) — rides on the same transactional email provider as OTP.
- **Status.** accepted
- **Owner.** Syle
- **Implication.** PRD §6 gets a "Build Order within MVP" subsection reflecting the above. Sprint planning can sequence epics directly off this. The realtime + local-first early build is the load-bearing piece — architecture must support it before §4.2 starts.
